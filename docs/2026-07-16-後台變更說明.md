## 一、範例輸出改為區塊陣列

### 改了什麼

後台表單的「範例輸出」原本只是一個純文字框，只能存一段文字。現在變成可增減的區塊，每個區塊各自選類型、可上下排序。

### 資料格式

```js
exampleOutput: [
  { type: "text",  data: { context: "【錯誤原因】…" }, seq: 0 },
  {
    type: "image",
    data: { context: "https://…/a.png", alt: "示意圖", caption: "gpt-image-2" },
    seq: 1,
  },
]
```

- `type`：`text`（文字）／`image`（圖片）／`video`（影片）／`html`
- `data.context`：必填。`text` 是文字內容；其餘三種是**網址**
- `data.alt`、`data.caption`：選填，**只有 image / video / html 有**。`text` 的 `data` 只有 `context`
- `seq`：不是使用者填的欄位，送出時依區塊排列順序自動生成（從 0 開始）

`html` 的 context 指向一個 HTML 檔案的網址，前台將以 iframe 嵌入。

### 表單操作

- 依類型切換欄位：`text` 顯示多行輸入框；`image` / `video` / `html` 改為單行「網址」，並多出「替代文字」與「圖說」兩個選填欄位
- 排序用上下箭頭（`↑` `↓`），刪除用 `✕`，底部有「+ 新增區塊」
- 整個範例輸出是選填的 —可以一個區塊都不加
- 有區塊時 `context` 必填；**不驗證網址格式**（刻意決定，讓填表的人自由貼）
- 切換類型時已填的內容會保留，不會因為手滑切錯就清空

### 為什麼圖片 / 影片是存網址

之後圖片和影片會改成由後端上傳檔案，但**上傳流程的產物就是網址** —選檔 → 上傳 → 後端回傳網址 → 存進 `context`。所以存網址是終點，不是過渡。

屆時要改的只有輸入控制項（把文字框換成上傳元件），`{ type, data, seq }` 這個資料形狀不用動，轉換邏輯與前台顯示也不用重寫。若當初存的是 base64 或檔案物件，才會整個打掉重來。

唯一可能影響形狀的變數：如果後端回傳的是**檔案 ID** 而非網址（好處是換 CDN 時舊資料不會失效），那 `context` 存的東西就要重新討論。這點值得跟後端確認。

### 相關檔案

  - `toBlocks()`：舊格式 → 表單用的區塊陣列（編輯模式載入時用）
  - `toPayload()`：送出前補 `seq`，並裁掉 `text` 不該有的 `alt` / `caption`

## 二、狀態改為布林（啟用 / 未啟用）

### 改了什麼

草稿 / 已發布 / 封存三種狀態，壓成一個布林：啟用或未啟用。草稿的概念不再需要。

### 順帶修掉一個既有的錯誤

探查後發現 `status` 不只是多餘，而且是錯的：

- **前台的可見性一直是由 `is_active` 決定**（`promptApi.js` 的 `getPublishedPrompts` 過濾的是它），跟 `status` 無關
- **mockData 的 seed 資料根本沒有 `status` 欄位**，只有 `is_active: true`
- 結果後台把所有 seed 進來的 prompt 都顯示成「草稿」（badge 找不到 status 就 fallback），但它們在前台其實全部是上架中 —**畫面上的狀態是騙人的**
- Dashboard 的「已發布 / 草稿」統計同理，兩個都顯示 0

改成布林後，這些假象一併消失。後台顯示的狀態現在是真的。

### 關鍵：`is_active` 必須雙寫

seed 資料的欄位是 snake_case 的 `is_active`，後台新增的則是 camelCase 的 `isActive`，兩種命名會同時存在。**前台讀的是 snake_case 那個。**

所以 `createSkill` / `updateSkill` 同時寫入兩個欄位：

```js
isActive: data.isActive ?? true,
is_active: data.isActive ?? true,  // 前台 getPublishedPrompts 讀這個
```

若只寫 camelCase，在後台按停用會**看起來成功但前台照樣上架** —badge 變了、前台沒變。這是最容易踩的坑，已寫專門的測試守住（停用 seed 資料後，它必須從 `getPublishedPrompts` 消失）。

讀取一律走 `isSkillActive()`，統一處理 `isActive ?? is_active ?? true` 的 fallback，不要在各處自己判斷。

### UI 變動

| 位置 | 原本 | 現在 |
|---|---|---|
| 表單「發布設定」 | 狀態下拉（三選一） | 滑動 toggle，「啟用中：會顯示在前台」／「已停用：不會顯示在前台」 |
| 列表狀態欄 | 三色 badge | 兩色：啟用（綠）／未啟用（灰） |
| 列表操作按鈕 | 「封存」 | 「停用」／「啟用」，依目前狀態切換 |
| 列表篩選 | 三選項 | 全部（預設）／啟用／未啟用 |
| Dashboard 統計 | 已發布 / 草稿 | 啟用 / 未啟用 |

停用會跳確認框（會影響前台）；啟用則直接執行。狀態不再是必填驗證項 —布林永遠有值。

### 相關檔案

- `src/api/adminApi.js` — 新增 `isSkillActive()`、`ACTIVE_OPTIONS`；`archiveSkill` → `setSkillActive(id, isActive)`；移除 `STATUS_OPTIONS`、`getStatusLabel`
- `src/api/skillActive.test.js`（新增）— 11 個測試
- `src/api/promptApi.js` — 移除傳給前台但無人使用的 `status`
- `src/components/admin/statusBadge.jsx`、`skillFilterBar.jsx`、`skillTable.jsx`、`skillForm.jsx`
- `src/pages/admin/AdminSkillsView.jsx`、`AdminDashboard.jsx`

舊資料殘留的 `status` 欄位不再被任何程式碼讀取，不需要遷移。

## 已知未完成（刻意）

以下都是討論後決定不做的，**不是漏掉的 bug**：

### 1. 前台看不到新的範例輸出

`src/pages/prompt/skillDetail.jsx` 仍讀舊的 `outputText` / `outputImages`，所以用新表單存的區塊資料在前台不會顯示。舊資料照常顯示。

### 2. mockData 的 snake_case 不一致

seed 資料是 `example_output`、`prompt_content`、`content_type_id` 等底線命名，表單讀的是駝峰命名，兩邊對不上。造成的實際後果：

- 編輯任何一筆 seed 資料時，主要內容、範例輸入、資料類型、所屬分類全部是空的
- 資料類型與所屬分類是必填，所以編輯 seed 資料按儲存會被驗證擋下 —**改不動**
- 連帶讓 seed 資料的啟用狀態只能從列表按鈕切，編輯頁的 toggle 對它們無效（對後台新增的資料則完全可用）

**這是既有問題，不是這兩次改動造成的。** 決定不修，因為 mockData 之後會由其他人整個換成新的資料格式，屆時這個不一致會自然消失 —現在加映射層等於為即將被刪除的東西寫轉接程式。

## 給之後接手的人

等 mockData 換成新格式後，以下兩處可以直接收掉，程式碼裡都有註解說明原因：

- `exampleOutputBlocks.js` 的 `toBlocks()` 中處理舊 `{ outputText, outputImages }` 格式的分支 → 會變成死碼
- `adminApi.js` 的 `isActive` / `is_active` 雙寫，以及 `isSkillActive()` 的 fallback → 前台不再讀 snake_case 就不需要了
